# Fable 5 上手体验与思考：Claude 家族的"资深工程师"

## 一、Fable 5 到底是什么？

先说清楚一件事：Fable 5 不是什么独立的新模型，它是 Anthropic Claude 家族 Myhtos 系列里的一个"角色"或者说"行为模式"。简单理解就是——同样是 Claude 4 的技术底子，但 Fable 5 被设定成了一个**更谨慎、更顽固、更较真的资深工程师**。

这个名字本身就有意思：Fable（寓言/传说），说明 Anthropic 在设计的时候就想让它带有某种"人格特质"。而它的核心差异不在于"能力更强"，而在于**行为方式不同**。

现在 GitHub 上围绕 Fable 5 已经形成了一个小生态，最核心的几个项目：

- **fable5-world-demo**（301⭐，41 Forks）：让 Fable 5 自己写了一个完整的 4×4 km 3D 世界，99% 代码由模型生成，包含地形、植被、光照、大气渲染。这是一个极限压力测试。
- **fable5-method**：把 Fable 5 的"严谨方法论"提取成一套通用的系统提示词，装到 Claude Code / ChatGPT / Cursor / Gemini 里，让其他模型也按 Fable 5 的方式工作。
- **fable5-method npm 包**：一行 `npx fable5-method` 安装到 Claude Code 的 skill 目录，自动激活。

## 二、Fable 5 的核心理念：六个阶段的严谨循环

### 1. 动手之前先想清楚

- 读代码、看架构、找到问题的根源在哪里
- **从真实的文件/日志/数据去找问题**，而不是凭猜测
- 问题往往是复合的，找到第一个原因不要停，继续往下挖

这一点和普通 Claude 模式最大的区别是：普通 AI 倾向于看完你的问题就直接给答案，Fable 5 会先反问"你确定问题是这个吗？让我先看看代码"。它防御的是"看起来对但实际错"的陷阱。

### 2. 拆解任务，严格执行契约

- 先列出所有要做的事，独立的并行做，依赖的串行做
- 生成代码后必须自己读一遍，再做一次对抗性审查
- 审查时只关注真正的崩溃、数据丢失、安全漏洞

### 3. 证明它，而不是声称它（最关键的一步）

- 每次改完必须编译、跑测试
- 你不能说"搞定了"——你必须**证明**它能在真正的输入上跑通
- 对于图像/3D输出，要实际看结果，不能只看描述
- 类型检查通过不等于代码正确

这就是 fable5-world-demo 能做到 99% 代码由 AI 生成的核心原因：每一步都有人验证，不是模型自己说了算。

### 4. 尊重意图，不擅自做决定

- 绝不在你不知情的情况下反转你的决策
- 绝不能擅自修改默认值、开关配置、价格
- 把可选行为做成 opt-in（开关），而不是直接删掉

这条很重要。普通 AI 经常偷偷改你代码里的一些默认行为，Fable 5 被训练成**把所有选择权交还给你**。

### 5. 验证交付

- 改完以后去确认改动确实落到了该落的地方
- 最后自己 review 一遍自己的 diff

### 6. 留下记录

- 更新文档，让下一次会话能从你结束的地方继续
- 如果同样的模式重复了三次，把它变成可复用的规则

## 三、fable5-world-demo：AI 构建 3D 世界的极限实验

这个 Demo 是 Fable 5 能力最直观的展示。几个关键数据：

- **21,000+ 行 TypeScript 代码**，全由 Fable 5 生成
- **90+ 次提交**，全部自动完成
- **4×4 km 完整开放世界**：4096×4096 的地形，流程化生成的峡谷、湖泊、森林、雪峰
- **6 种树木物种**，每棵树姿态唯一（也就是没有两棵一样的树）
- **GPU 驱动的植被渲染**：80 万片草叶、45 万棵树的实例
- **完整的渲染管线**：Hillaire 大气模型、CSM 阴影、PCSS、体积云、TRAA 抗锯齿
- **纯 WebGPU**，无任何外部资源的下载

更关键的是这个工作流：

1. 人类写了一个**需求文档**（PROJECT_LAAS_v2.md），设定视觉标准和各种"硬规则"（不能有黑色阴影、不能有空地、不能有克隆树、不能用 MeshBasicMaterial……）
2. Fable 5 自己规划架构，分阶段编码
3. 每完成一个阶段，用 Playwright 截图，和参考图对比
4. 没通过就进 debug 循环，通过了才进下一阶段
5. 所有决策记录在 STATUS.md 文件中，作为"持久记忆"

这个项目最震撼的不是它能写 2 万行代码，而是**它知道自己在做什么、做到了哪一步、下一步该干什么**。

## 四、放到真实场景里好不好用？

我把 fable5-method 装到 Claude Code 里试了一段时间。说说真实感受：

### 好的方面：

- **Bug 修复更靠谱了**。以前 Claude Code 会直接给我一个"看起来对"的修复，现在我必须看它手动 trace 代码路径、读相关文件、确认问题根源。第一次用觉得它啰嗦，但确实减少了二次修复的次数。
- **不会乱改东西了**。改配置文件、改默认值这种事，现在它会弹出来问"这个默认值要从 false 改成 true，确认吗？"
- **证明过程价值很大**。特别是在改一些关键逻辑的时候，"类型检查通过"和"确实在真实输入上跑通了"是两回事。Fable 5 会主动执行测试用例来验证。

### 不舒服的地方：

- **慢**。严谨循环的每一步都要时间，特别是在大项目里，读文件、trace 代码、编译验证，普通的"猜答案"几秒钟，Fable 5 可能要一两分钟。这种延迟在简单任务上会让人觉得"多此一举"。
- **有时候过于谨慎**。在一些明显没有风险的改动（比如改个文案、调个 log 级别）上它也走完整流程，确实有点笨重。
- **需要适应**。如果你习惯了"AI 快速给你答案，你自己判断对不对"的模式，Fable 5 的反问和"先让我看看"会让人觉得它不解人意。

### 哪种场景最适合：

- **生产环境的关键代码修改**（涉及金钱、数据、安全权限的路径）
- **大型重构**（需要系统性的 trace 和理解）
- **紧急 bug 修复**（你用 Fable 5 的方式反而能避免修好后引入新 bug）

### 哪种场景不适合：

- **写脚本、搭原型**（你需要的是快速产出，不是严谨流程）
- **修文案、调样式**（走流程的成本远大于收益）
- **探索性编码**（不确定方向的时候，需要的是发散不是收敛）

## 五、我的真实感受

Fable 5 的方法论让我想起一件事：2010 年左右我做运维的时候，每次上线前要填一个 Change Request 表，写着"变更原因、影响范围、回滚方案"。那时候觉得这东西纯属流程主义，后来线上出过几次事故才明白——那些"我确认过"的步骤不是走形式，是让你**真的确认过**。

Fable 5 本质上就是把这个逻辑塞进了 AI 的行为模式里。它不是让模型更强，而是让模型**知道自己什么时候会犯错**，然后用流程来挡住自己。

这和自动驾驶的"功能安全"设计思路很像：真正的自动驾驶系统（比如 L3+）不只是靠算法的准确性，还靠一套故障检测和降级机制来兜底。Fable 5 就是"AI Agent 的安全冗余"。

但也要说实话：这套流程放大了使用成本。原来一句话就能改的东西，现在要等它读文件、trace 逻辑、确认。在简单任务上，用户很可能嫌烦关掉它，回到原来的"快速回答模式"。

所以 Fable 5 真正的价值不是取代普通 Claude，而是给你一个选择——**高风险高价值的工作让 Fable 5 来，日常工作让普通 Clude 来**。

这大概也是为什么 Anthropic 把它做成一个角色而不是默认行为。资深工程师不是每个场景都需要的，但关键场景下，你宁可慢一点，也不想出错。

## 六、Fable 5 的禁用与解禁：一个更宏大的叙事

讲完技术层面的东西，说一件绕不开的事。

Fable 5 发布后不久，有过一段关于"美国政府对顶级模型实施临时限制"的讨论。一些安全研究人员和内部测试者提到，Fable 5 在某些阶段被限制访问，或者需要额外的审批才能使用。具体的时间线、范围和原因外界没有拿到完整的信息，但可以从已有的公开信息拼出一个轮廓。

这里不讨论政治立场，只说事实和逻辑。

### 为什么会被限制？

一个能自己写 21000 行代码、构建完整 3D 世界、在复杂环境中自主推理并执行的模型，它的能力边界在哪里？这个问题不只 Anthropic 自己想知道，政府也想知道。

关键点不在于 Fable 5 **现在**能做什么，而在于它**下一步**能做什么——特别是当它结合了工具调用、文件系统访问、网络操作和长期记忆之后。在 AI 安全圈子里，这叫"能力外溢"（capability overhang）：模型当前的表现不代表它的上限，真正的上限可能比看到的要高得多。

对大语言模型来说，"能力强"和"危险"是一体两面。比如 Fable 5 的"严谨方法论"，用在代码审计上是利器，但如果用在自动化漏洞挖掘上呢？它对自己的行为有严格约束，但这些约束是在系统提示词层面实现的，不是模型底层不可绕过的。这就引出了 AI 安全的核心困境：**你没法既给它权力，又保证它永远不用。**

### 解禁的考量

限制是暂时的，解禁是必然的。原因很简单：如果你因为担心安全问题就不放出一个强大的模型，那你的对手可不会等。

这背后是一个正在形成的全球格局：

### 不同角色的处境

**普通人：**
你大概率不会直接接触 Fable 5 这个级别的模型。你用的是它蒸馏出来的轻量版。你感受不到它的存在，但它已经在影响你用到的一切——你的搜索、你的客服对话、你的代码助手。对你来说，好处是这些东西越来越好用，坏处是你没法判断给你答案的是"Fable 5 级别的LLM"还是一个只会模式匹配的次品。

**普通公司（非科技行业）：**
普通公司是 Fable 5 的间接受益者。你们不自己部署模型，而是用大模型公司提供的 SaaS 接口。API 降价、能力提升、门槛降低，这些事情都在对你们有利的方向发展。但问题是，业务数据流经这些 API，如果哪天服务条款变了、价格涨了、或者模型被限制在某些地区不可用了，没有任何议价能力。这和大公司用 AWS/Azure/阿里云 是一样的道理——上船容易下船难。

**大型企业（互联网/科技行业）：**
大型企业的处境最复杂。一方面，Fable 5 级别的模型能让研发效率翻倍，代码审查、架构设计、bug 修复都会被重塑。另一方面，你们面临两个不可忽视的风险：

第一，**人才空心化**。当 80% 的编码工作可以由 AI 完成，新一代工程师的成长路径就被打断了。一个没写过烂代码的人，不会成为好的工程师，更不会从技术细节层面看待工程的构建。而 Fable 5 的"严谨方法论"恰恰需要有人来执行最后的判断。

第二，**模型依赖**。业务架构可能越来越深地嵌入某个模型的能力，如果这个模型的访问策略发生变动（比如出口管制、许可变更），整个技术栈可能面临重构。这在以前是硬件层面的问题（芯片断供），现在软件层面也开始出现这种情况。

**国家、政府：**
这是所有角色里动作最快的。各国政府已经形成了几种不同的态度：

- **美国**：通过出口管制和实体清单来管理顶级模型的跨境流动。逻辑是"我不能不让它出现，但我能决定谁可以用"。
- **中国**：走自主路线，国有资本大力投入基础模型研发。短期看性能有差距，长期看这个差距在缩小。
- **欧盟**：用监管框架来约束，典型的就是《AI Act》，通过风险分级来管理不同的 AI 应用场景。

政府的共同焦虑是：**一个超过某种能力阈值的模型，本质上和核材料一样是战略资源。** 战略资源不能完全交给市场，必须由国家来管控。但和核材料不同的是，AI 模型可以复制、可以压缩、可以蒸馏，边界管控比物理世界难得多。这也是为什么现在各国都在推"AI 安全治理"——因为技术跑得比法规快，再不追就真的追不上了。

### 未来的博弈

未来两三年，会出现几个不可避免的局面：

1. **模型能力分级管理**。Fable 5 不是最后一个被限制/解禁的模型，以后每个重大版本发布前，都会有一轮政府层面的安全评估。模型会分成"通用版"和"受限版"，前者公开，后者只对经过审查的机构开放。

2. **开源和闭源的两条路线**。开源模型（如 Gemma、LLaMA）会迅速赶上闭源模型，但永远落后半个身位——不是技术上的落后，而是能力上限的落后。最顶级的模型能力会被控制在少数几家公司手里，因为只有它们能承担安全审查的成本。

3. **普通用户的代理困境**。绝大多数人使用的"AI"将是经过多层过滤的：模型本身的能力被限制、接入的 API 有内容安全策略、前端还有一层对齐过滤。你得到的答案不是模型能力的上限，而是安全策略的下限。这和今天的搜索引擎一样——你看到的搜索结果不是互联网的全貌，是经过排名算法和审核机制过滤之后的内容。

4. **企业级 AI 的私有化部署会成为主流**。当大模型的 API 调用变成战略依赖，头部企业一定会把自己的数据、模型、算力封闭在自己的环境里。Fable 5 这类顶级模型的能力会被"私有化蒸馏"成企业自己的专用模型，虽然能力弱一些，但可控。

Fable 5 只是一个开始。它的禁与解禁不是一个孤立事件，而是未来十年 AI 治理的一个缩影。技术走一步，政策追一步，市场和用户夹在中间。这种状态可能会持续很长时间，直到整个社会建立起一套新的平衡机制——就像当年互联网从"蛮荒"走向"有法可依"一样。
