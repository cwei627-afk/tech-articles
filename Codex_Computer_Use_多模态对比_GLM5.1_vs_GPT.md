# Codex 接入国产 GLM-5.1 vs 原生 GPT-5.4/5.5：Computer Use 多模态能力实测

最近在 Codex 里测试了国产 GLM-5.1（多模态）和原生 GPT-5.4/5.5 的 Computer Use 功能，发现了一些有意思的东西。借此聊聊国产多模态和美国多模态的真实差距。

先说结论：除去原生 Agent 与模型适配的因素，GLM-5.1 的多模态能力在 Computer Use 场景下确实有明显短板。网页操作填写表单和键鼠操作这两项能力，和国外的 GPT-5.4/5.5 有显著差距。但有意思的是，GLM-5.1 在多模态数据的输入和分析这些基础功能上，做得其实还不错。

---

## Computer Use 是什么？

简单说，就是让 AI 模型直接操控你的电脑。包括：
- 屏幕理解（视觉）
- 键盘鼠标操作（动作）
- 网页交互（理解 UI + 填表单）

这是检验多模态能力的终极场景——模型不仅要"看懂屏幕"，还要"做出正确操作"。比单纯的图像理解难多了。

Codex 的 Computer Use 实现方式是通过 Desktop Touch MCP（基于 Rust 引擎 + Windows UI Automation），把屏幕截图喂给模型，模型返回操作指令（点击坐标、键盘输入、滚动等），然后执行。

---

## 测试环境和配置

**GLM-5.1 接入方式：**
- 通过 OpenClaw Gateway 桥接 Codex API
- 模型配置：`glm-5.1-heretic`，temperature 0.1
- 视觉输入：Base64 编码的 PNG 截图（1920x1080）
- 响应格式：自定义 JSON Schema（action_type + coordinates + input_text）

**GPT-5.4/5.5：**
- Codex 原生支持，无需桥接
- 内置 Computer Use 工具链，直接返回结构化操作指令
- 后端集成 Windows UI Automation API

**测试环境：**
- Windows 11 Pro 22621
- Chrome 126.0.6478.127
- Desktop Touch MCP v1.10.4
- 截图分辨率：1920x1080，PNG 格式

## 测试任务设计

我设计了 5 个典型场景，覆盖网页操作和键鼠操作两类：

### 网页表单类

**任务 1：GitHub Issue 提交**
- 打开 github.com/new
- 填写 Title："Bug: Login button not working on mobile"
- 选择 Label："bug"
- 填写 Description："When I click login on iPhone Safari, nothing happens"
- 点击 Submit

**任务 2：问卷星调查问卷**
- 打开某问卷星链接
- 填写 5 道单选题 + 2 道填空题
- 提交问卷

### 键鼠操作类

**任务 3：记事本编辑**
- 打开记事本
- 输入一段 50 字的文本
- 选中前 10 个字，删除
- 保存文件到桌面

**任务 4：Windows 文件操作**
- 打开资源管理器
- 导航到 D:\Project
- 找到并打开某个文件夹
- 复制一个文件到桌面

**任务 5：多窗口切换**
- 同时打开 Chrome 和 VS Code
- 在 Chrome 中搜索一段代码
- 切换到 VS Code 粘贴搜索结果
- 保存文件

## 测试结果

每个任务测试 10 次，记录成功率、平均耗时、失败原因：

| 任务 | GLM-5.1 成功率 | GPT-5.4 成功率 | GPT-5.5 成功率 | GLM-5.1 耗时 | GPT-5.4 耗时 | GPT-5.5 耗时 |
|------|---------------|---------------|---------------|-------------|-------------|-------------|
| GitHub Issue | 0/10 | 8/10 | 9/10 | 45s (失败) | 18s | 22s |
| 问卷星 | 1/10 | 7/10 | 9/10 | 52s (失败) | 25s | 30s |
| 记事本编辑 | 2/10 | 9/10 | 10/10 | 38s | 12s | 15s |
| 文件操作 | 1/10 | 8/10 | 9/10 | 60s (失败) | 20s | 24s |
| 多窗口切换 | 0/10 | 6/10 | 8/10 | 90s (失败) | 35s | 42s |

**总体成功率：**
- GLM-5.1：4/50 = 8%
- GPT-5.4：38/50 = 76%
- GPT-5.5：45/50 = 90%

差距明显。GPT-5.5 最稳，GPT-5.4 略快但偶尔翻车。GLM-5.1 基本不能完成任务。

## GLM-5.1 的具体失败表现

### 坐标计算偏差大

实测发现 GLM-5.1 返回的点击坐标经常偏差 50-200px。

**例子：GitHub "Submit new issue" 按钮**
- 实际位置：(920, 680)
- GLM-5.1 返回：(850, 520)
- 偏差：70px 水平 + 160px 垂直
- 结果：点击到了页面空白区域

**分析原因：**
GLM-5.1 的视觉编码器在处理高分辨率截图时，可能没有经过足够的"坐标定位"训练。它能识别"有个按钮"，但像素级定位不准。

对比 GPT-5.4 的返回：(918, 682)，偏差不到 5px。

### 操作序列断裂

GLM-5.1 经常在多步骤操作中"断片"。

**问卷星任务的典型失败：**
```
步骤1: 填写姓名 → 成功（运气好）
步骤2: 选择第1题选项 → 失败（坐标偏差）
步骤3: 返回的 JSON 里 action_type 变成了 "unknown"
步骤4: 后续步骤全部丢失
```

GPT-5.4/5.5 会明确返回每一步的状态，失败后还能尝试修正。

### 屏幕元素识别但不理解语义

GLM-5.1 能识别"这是一个输入框"，但不知道"这是用来填写邮箱的"。

**例子：登录页面**
- 有用户名和密码两个输入框
- GLM-5.1 把密码填到了用户名框
- 因为它只识别了"输入框"，没理解字段的语义

GPT-5.4/5.5 会读取 placeholder 文案、label 标签，理解"哪个框填什么"。

### 多窗口场景完全失灵

任务 5（多窗口切换）GLM-5.1 全部失败。

原因分析：
- 截图只能看到一个窗口
- GLM-5.1 不知道另一个窗口在哪
- 没有"窗口列表"的概念
- 无法生成 Alt+Tab 切换指令

GPT-5.4/5.5 内置了窗口管理 API 调用能力，可以枚举窗口、获取窗口位置、执行切换。

## 但 GLM-5.1 的视觉理解能力不错

单纯测试"看图说话"能力，GLM-5.1 表现其实挺好：

**测试：给一张网页截图，让模型描述页面内容**

GLM-5.1 的回答：
```
这是一个 GitHub 的创建 Issue 页面。
页面顶部有仓库名称 "user/repo-name"。
中间有两个输入框，一个是 Title，一个是 Description。
右侧有 Labels 选择区域，可以看到 "bug"、"enhancement" 等标签。
底部有一个绿色的 "Submit new issue" 按钮。
```

描述准确，元素识别正确。问题是它知道这些，但做不到"点击那个按钮"。

**结论：GLM-5.1 的视觉编码器没问题，是"视觉→动作"的映射层没训练好。**

## 差距的技术根因

### 原生适配 vs 桥接适配

一个关键差异：GPT-5.4/5.5 是 Codex 原生适配的，内置了 Computer Use 工具链：

```json
// GPT-5.5 返回的操作指令格式
{
  "tool": "computer",
  "action": "click",
  "coordinates": [920, 680],
  "element_description": "Submit new issue button",
  "confidence": 0.95
}
```

GLM-5.1 是通过 OpenClaw 桥接，需要把截图转成文本描述，然后让模型生成操作指令。中间多了一层转换，而且没有专门的"坐标定位"训练。

### 训练数据的差异

OpenAI 的 Computer Use 能力来自专门训练：
- 数百万次"截图→操作"配对数据
- 模型学会了"看到某个 UI 元素→输出对应坐标"
- 这是一种"视觉-动作映射"的训练

GLM-5.1 的训练数据可能更偏重：
- 图像理解（"这是什么"）
- 文本生成（"描述这张图"）
- 缺少"坐标定位"和"操作序列规划"的训练

这不是模型架构的问题，是训练目标的问题。

### OpenAI 方案的局限性

顺便说一下，OpenAI 的 Computer Use 方案其实也不是"真正的多模态"：

它的实现方式：
1. Windows UI Automation API 获取元素树
2. 截图 + 元素树一起喂给模型
3. 模型输出操作指令（包括坐标）
4. API 执行操作

这是一种"工程化方案"。模型被训练成：
- 识别屏幕元素
- 计算坐标
- 调用底层 API

优点：精准可控，Windows 上效果很好
缺点：依赖微软 API，跨平台难，Linux/macOS 需要重新适配

## DeepSeek 在做真正的多模态

DeepSeek 的探索方向不太一样，他们的方案更像"原生多模态"：

### 技术路线

根据公开的技术讨论，DeepSeek 的思路是：

**不依赖坐标计算**
模型直接输出语义级操作：
```json
{
  "action": "input_text",
  "target": "login_form.username_field",
  "value": "user@example.com"
}
```

而不是：
```json
{
  "action": "click",
  "coordinates": [245, 320],
  "input": "user@example.com"
}
```

**端到端视觉-动作学习**
训练数据是"截图→操作意图"配对，而不是"截图→坐标"配对。模型学习的是"看到登录框→知道要填用户名"，而不是"看到登录框→计算它的像素位置"。

**抽象操作层**
定义一套跨平台的操作语义：
- `click(element_id)` 而不是 `click(x, y)`
- `input(field_selector, text)` 而不是 `type_at_position`
- 这套语义可以映射到不同平台的底层 API

### 方案对比

| 方案 | OpenAI 主流 | DeepSeek 探索 |
|------|------------|---------------|
| 定位方式 | 像素坐标计算 | 元素语义选择器 |
| 平台依赖 | 强（微软 UIA API） | 弱（抽象层适配） |
| 训练目标 | 坐标回归 | 操作意图预测 |
| 跨平台 | Windows 优先，其他需重训练 | 一套模型跨平台 |
| 精度来源 | API 辅助 + 坐标计算 | 模型语义理解 |
| 成熟度 | 已商用 | 探索阶段 |

DeepSeek 的路线更像"让模型像人一样理解界面"。人不需要知道按钮的精确坐标，只需要知道"点击那个提交按钮"。

## 对国产多模态的思考

### 差距要承认

Computer Use 是多模态能力的试金石。GLM-5.1 在"理解→行动"链条上确实有断点：

- 视觉理解层：没问题，能准确描述屏幕内容
- 操作规划层：缺失，不知道怎么把理解变成指令
- 坐标定位层：薄弱，偏差经常超过 50px
- 序列执行层：不稳定，多步骤容易断裂

这不是简单的"适配问题"，是训练目标和训练数据的差异。

### 两条追赶路径

**路径 1：工程化追赶（短期）**

参照 OpenAI 方案：
- 训练专门的"坐标定位"模型
- 集成平台底层 API（Windows UIA）
- 用 API 辅助提高精度

优点：3-6 个月可能见效
缺点：天花板是 OpenAI 的水平，而且平台依赖重

**路径 2：原生多模态突破（长期）**

像 DeepSeek 一样，探索语义级多模态：
- 不依赖坐标，用元素选择器
- 端到端训练"截图→操作意图"
- 跨平台架构

优点：可能弯道超车，不依赖微软 API
缺点：投入大，周期 1-2 年，风险高

### 建议

**短期（6 个月）：**
工程化补课。先让 GLM 在 Computer Use 上"能用"，哪怕精度不如 GPT。

**中期（1 年）：**
强化训练数据。收集"截图→操作"配对数据，训练"视觉→动作"映射能力。

**长期（2 年+）：**
布局语义级多模态架构。如果 DeepSeek 的路线验证成功，国产厂商应该跟进。

## 最后

GLM-5.1 和 GPT-5.4/5.5 在 Computer Use 上的差距，其实反映了国产多模态的真实水平：

- **输入端（看）**：已经追上了，视觉理解能力没问题
- **输出端（做）**：还有代差，坐标定位和操作规划薄弱
- **工程适配**：有影响，但不能把差距全甩锅给"不是原生适配"

好消息是，差距的根因找到了——是训练目标和训练数据的问题，不是架构问题。这意味着有追赶路径。

DeepSeek 的探索让我看到了另一种可能——真正的多模态不只是"看得准"，更要"做得对"。不需要像素级坐标，只需要语义级理解。这条路线如果能跑通，也许是国产多模态弯道超车的机会。
